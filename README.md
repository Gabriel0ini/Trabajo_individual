# Resumen: Git básicoGabriel Jhoan Cespedes Garcia## ¿Qué es Git?Sistema de control de versiones distribuido (VCS) que permite guardar y gestionar cambios en archivos a lo largo del tiempo de forma local.## ¿Cómo nació Git?Git fue creado en 2005 por Linus Torvalds durante el desarrollo del kernel de Linux, cuando la comunidad perdió acceso a la herramienta que utilizaban antes, llamada BitKeeper. Ante la necesidad de seguir coordinando el trabajo de muchos desarrolladores y no depender de software propietario, decidió crear su propio sistema de control de versiones,que lo creó en 2 o 3 semanas que se quedo desarrollando en su casa.Diseñó Git para que fuera rápido, distribuido y seguro, permitiendo que cada desarrollador tuviera una copia completa del proyecto y pudiera trabajar de manera eficiente sin depender de un servidor central.## ¿Cómo instalar Git?- Ir a: https://git-scm.com/install/- Seguir pasos según el sistema operativo Linux o Windows- Verificar instalación con:  git --version## Configuraciones básicas- git config --global user.name "Gabriel Jhoan Cespedes Garcia"- git config --global user.email "gabrijhhoan2006@gmail.com"## Archivos esenciales en un repositorio- README.md → descripción del proyecto- .gitignore → archivos que Git debe ignorar# States y Commits — Git SCESI Clase 2## ¿Qué es esto?Resumen de la clase 2 del curso de Git SCESI 2026,donde vimos los estados de Git y cómo escribir buenos commits.## Los 3 estados de Git**1. Directorio de Trabajo**Donde escribís tu código. Git ve tus archivos como:- `Untracked` → archivo nuevo- `Modified` → archivo que ya existía y se modifique```bashgit restore       # revertir cambios echo "archivo" >> .gitignore  # ignorar un archivo```**2. Stage Area**Seelige  qué cambios van en el próximo commit.```bashgit add    # agregar un archivogit add .           # agregar todogit restore --staged   # sacarlo del stage```**3. Repositorio Local**Tus cambios quedan guardados en el historial con un hash único.```bashgit commit -m "mensaje"      # guardar cambiosgit reset --soft HEAD~1      # deshacer último commit``## Buenas prácticas- Hacé commits chicos y con sentido (**commits atómicos**)- Que cada commit no rompa el proyecto- Máximo 50 caracteres en el mensaje- Sin punto final```bash# malgit commit -m "Fix things."# bien  git commit -m "fix: Resolve login error on mobile"```## Prefijos| Prefijo | Cuándo usarlo ||---------|---------------|| `feat` | Nueva funcionalidad || `fix` | Corrección de bug || `docs` | Documentación || `style` | Formato del código || `refactor` | Reestructurar código || `test` | Agregar o arreglar tests || `perf` | Mejorar rendimiento |

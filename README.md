Directorio de Trabajo**Donde escribís tu código. Git ve tus archivos como:- `Untracked` → archivo nuevo- `Modified` → archivo que ya existía y se modifique```bashgit restore       # revertir cambios echo "archivo" >> .gitignore  # ignorar un archivo```**2. Stage Area**Seelige  qué cambios van en el próximo commit.```bashgit add    # agregar un archivogit add .           # agregar todogit restore --staged   # sacarlo del stage```**3. Repositorio Local**Tus cambios quedan guardados en el historial con un hash único.```bashgit commit -m "mensaje"      # guardar cambiosgit reset --soft HEAD~1      # deshacer último commit``## Buenas prácticas- Hacé commits chicos y con sentido (**commits atómicos**)- Que cada commit no rompa el proyecto- Máximo 50 caracteres en el mensaje- Sin punto final```bash# malgit commit -m "Fix things."# bien  git commit -m "fix: Resolve login error on mobile"```## Prefijos| Prefijo | Cuándo usarlo ||---------|---------------|| `feat` | Nueva funcionalidad || `fix` | Corrección de bug || `docs` | Documentación || `style` | Formato del código || `refactor` | Reestructurar código || `test` | Agregar o arreglar tests || `perf` | Mejorar rendimiento |# Resumen Clase 3: GitHub y SSHEste repositorio contiene los apuntes y prácticas de la sesión sobre gestión de repositorios remotos y configuración de seguridad.## 1. Los cambios realizados en este estado **se pierden** al salir, a menos que se creen en una rama nueva:```bashgit checkout <hash_antiguo>       # Ir al pasadogit checkout <rama>               # Volver al presentegit checkout -b rama_nueva        # Rescatar cambios del Detached HEAD```### Buenas prácticas- Haz commit de tu trabajo antes de navegar entre commits.- Evita trabajar mucho tiempo en estado Detached HEAD; mejor crea una rama.- Aprovéchalo para estudiar la evolución de proyectos grandes.

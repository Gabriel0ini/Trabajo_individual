Los cambios realizados en este estado **se pierden** al salir, a menos que se creen en una rama nueva:```bashgit checkout <hash_antiguo>       # Ir al pasadogit checkout <rama>               # Volver al presentegit checkout -b rama_nueva        # Rescatar cambios del Detached HEAD```### Buenas prácticas- Haz commit de tu trabajo antes de navegar entre commits.- Evita trabajar mucho tiempo en estado Detached HEAD; mejor crea una rama.- Aprovéchalo para estudiar la evolución de proyectos grandes.# Clase 5: Ramas y GitFlow Básico## ¿Qué es una rama?Una línea de desarrollo independiente que permite trabajar en paralelo sin afectar el código principal.---## git branch| Comando | Acción ||---|---|| `git branch` | Lista todas las ramas y muestra dónde está el HEAD || `git branch <nombre>` | Crea una rama nueva desde la posición actual || `git branch -D <nombre>` | Elimina una rama |---## Moverse entre ramas```bashgit checkout <rama>      # Cambia a esa ramagit checkout -b <rama>   # Crea la rama y cambia a ellagit switch <rama>        # Alternativa moderna (solo ramas)```> `git switch` es más seguro porque no puede dejarte en Detached HEAD accidentalmente.---## GitFlowSistema de reglas para nombrar y organizar ramas en equipo.### Ramas principales- **main** → código en producción, siempre estable- **develop** → integración diaria del equipo, código en prueba### Ramas de apoyo| Rama | Nace de | Se fusiona en | Para qué ||---|---|---|---|| `feature/*` | develop | develop | Nueva funcionalidad || `release/*` | develop | main y develop | Preparar una versión || `hotfix/*` | main | main y develop | Parchar un error urgente |# Clase 6: Merge, Fetch, Pull y Push## Comandos clave### git merge --no-ffCombina los cambios de una rama dentro de otra. El flag `--no-ff` es importante porque obliga a crear un commit de fusión, lo que preserva el registro de que existió esa rama aunque después la elimines.### git fetchSolo "espía" el repositorio remoto para ver si hubo cambios, pero no modifica nada en tu código local. Es útil para saber si alguien más subió algo antes de descargar.### git pullDescarga los cambios remotos y los integra directamente a tu rama local. Es básicamente un `fetch` + `merge` en un solo paso.### git pushEnvía tus commits locales al repositorio remoto. La primera vez que subes una rama nueva debes usar `-u` para que Git registre la conexión entre tu rama local y la remota:```bashgit push -u origin mi-rama```---## Flujo de trabajo en equipo```bash# 1. Actualizar develop antes de empezargit checkout developgit fetchgit pull origin develop# 2. Ir a tu rama y sincronizarla si develop cambiógit checkout mi-ramagit merge develop# 3. Trabajas normalmente y subes tus cambiosgit push origin mi-rama       # Primera vez: git push -u origin mi-rama# 4. Integrar tu rama a developgit checkout developgit fetchgit pull origin developgit merge --no-ff mi-rama     # Si hay conflictos, resuélvelos manualmentegit add .git commit                    # Se abre el editor: Ctrl+O → Enter → Ctrl+X (nano)# 5.
